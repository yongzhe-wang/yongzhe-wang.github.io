Giving Claude the keys to my iOS Calendar

April 2026

  • GitHub repo

I wanted Claude to put things on my iPhone calendar. Apple doesn't have a remote iOS Calendar API I can just call. So I built a small MCP server that bridges Claude to my Mac's Calendar.app via AppleScript, and let iCloud do the rest. I called it apple-calendar-mcp.

The problem

iOS Calendar has no public remote API. EventKit is device-local. CalDAV works but is painful and requires credentials I'd rather not hand to a model.

But my iPhone and my Mac are on the same iCloud calendar. Anything that lands in Calendar.app on my Mac shows up on my phone within seconds.

So the real question wasn't “how do I reach iOS from Claude” — it was “how do I reach Calendar.app from an MCP server, cleanly.”

The approach

MCP server running locally on my laptop. Six tools: list_calendars, list_events, search_events, create_event, update_event, delete_event.

The transport under all of them is osascript driving Calendar.app. No network. No cloud. No credentials. macOS's own permission prompt is the only access gate.

How it works

Each tool is a thin TypeScript function that builds an AppleScript string, escapes every user input, runs it via osascript, and parses the result.

Record separator 0x1E, unit separator 0x1F for output parsing — not a security boundary, just a way to avoid CSV hell.

Dates never use AppleScript's locale-sensitive date “Monday, April 21, 2026 at 10:00:00 AM” literal. I construct dates by adding epoch seconds to a fixed January 1, 1970 anchor, which Calendar.app parses the same on every locale.

Event IDs are Calendar.app's own uid property. Stable across app restarts, survives moving between calendars in the common path, and lets me round-trip events through Claude without inventing synthetic keys.

What I learned

AppleScript escaping is security work. Every string I pass — event title, notes, calendar name, URL — goes through one sanitizer that backslash-escapes \ and " and wraps in quotes. Without it, a malicious event title could inject arbitrary AppleScript. I wrote the escape function first and a test for it second, before I wrote a single tool.

stdout is the transport. MCP speaks JSON-RPC on stdout. One errant console.log and the protocol breaks. Every log I write goes to stderr. This is a mistake I made once. I will not make it again.

Calendar.app is older than me. Accessing location or description on an event that doesn't have one throws. Setting a calendar by number works in some locales and not others. AppleScript is a language where half the bugs feel like folklore.

The worst bug was update_event. If you both rename an event and move it between calendars in the same call, Calendar.app helpfully saves the rename, then loses the reference before the move completes, and then quietly deletes the event with no trace. I found this by creating six calendar entries, trying to move them to the right calendar, and watching all six vanish. The fix: two separate operations, rename first, then move. The lesson: AppleScript errors lie, and the only trustworthy verification is a second read after the write.

Tests are the project's spine. I can't unit-test Calendar.app, but I can unit-test every string builder and every parser as a pure function, and I can slam adversarial payloads at the escape function. The tests are the reason I trust the server when it tells Claude it created an event.

What's next

Publish the repo under MIT.

Add Reminders.app support (Apple calls it Reminders, which is a separate AppleScript target).

Better recurring-event handling — right now I fetch the series and filter client-side, which is fine at my scale and painful at someone else's.

A small CLI so the server is useful even without an MCP client.
